# REQUIREMENTS — the locked / challengeable ledger

The **single source of truth** for this project's durable requirements. CLAUDE.md
invariants and [docs/ROADMAP.md](./ROADMAP.md) reference entries here **by ID** instead
of restating them — this file is where a requirement's *status* lives. *(The 2026-07-11
ground-up rebuild collapsed the four plugins into one `companion`; entries touched by it
are annotated below — see **R24**.)*

*Organized **chronologically by R-ID** (this ledger's native axis — it's decision history).
It sits at the centre of the shared spine: the contract (`docs/flows/`, `docs/INVARIANTS.md`)
references **this** file **by ID**, not the reverse — so there are no per-row UX
links here (that direction would duplicate the pillars, against R2 one-canonical-home).*

**Status vocabulary**

- 🔒 **locked** — *changing* it needs explicit owner sign-off; but *challenging* it is
  **invited** (R52·3) — the recommendation loop should actively surface challenges to locked
  entries rather than treat them as settled. Silence-by-deference is itself a miss.
- 🔓 **open** — up for challenge; the loop may propose changing these on its own signal.
- ⚰️ **retired** — no longer in force; kept for history with a pointer to what replaced it.

**The rule (R5):** every recommendation and every task-conversion **cites the R-IDs it
touches and names any it would change** — a requirement is never retired silently, in
either direction.


**Density standard (R66/R3, 2026-07-22):** entries are Claude-consumed — dense spec, not
narrative: constraint · status · provenance+date · the load-bearing why · amends/composes
links. Blow-by-blow history lives in git, not in rows. R1–R31 predate the standard (left
as-is this pass — compact opportunistically when touched); R32+ conform.
